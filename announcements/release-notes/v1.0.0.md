# CXF 1.0 Release Announcement

## Official Announcement

We are proud to announce the release of **CXF 1.0**.

After extensive community review, multiple draft iterations, interoperability hardening, and governance refinement, CXF now enters its first stable release as a forensic container standard designed for long-term evidence preservation, independent verification, and multi-vendor implementation.

CXF 1.0 was built around a simple principle: **a small, deterministic core with clear evidentiary boundaries**.

That principle now results in a format that provides:
- a deterministic CBOR manifest as the single authoritative metadata source
- signed core structures and a clear root-of-trust model
- SHA3-256 as the normative verification truth
- signed verifier outputs for evidence-grade reporting
- tightly bounded bridge profiles for controlled interoperability transitions
- external long-term evidence paths that preserve the immutability of the core container

This release reflects the community’s commitment to technical clarity, forensic reproducibility, and long-term standard stability.

## What CXF 1.0 delivers

### A stable forensic core
CXF 1.0 defines a strict and intentionally minimal container core:
- deterministic manifest handling
- no silent fallback behavior
- strict reject semantics
- no reuse of reserved fields
- clear separation between immutable container data and external evidence artifacts

### Evidence-grade verification
The standardized CXF verifier output is a signed CBOR artifact. This gives the ecosystem a portable, machine-readable, and cryptographically bound report format for evidence review, quality assurance, and interoperability testing.

### Long-term extensibility without core bloat
Long-term receipts and recovery evidence are kept outside the immutable core. This allows CXF to remain stable while supporting long-horizon archival, migration, attestation, timestamping, and recovery workflows.

### Stronger governance
CXF 1.0 also introduces the governance shape needed for a durable standard:
- structured digest sunset decisions
- auditable bridge lifecycle records
- interoperability reporting with coverage expectations
- explicit boundaries between normative evidence and non-authoritative projections

## What is not in the 1.0 core

To preserve the stability of the first release, several topics are intentionally treated as post-1.0 extension work rather than last-minute core additions. These include richer recovery evidence profiles, expanded governance automation, and performance-oriented candidate paths such as BLAKE3.

BLAKE3 is **not** part of the normative CXF 1.0 evidentiary core. It remains a possible future candidate path for strictly bounded performance use cases.

## Why this matters

For many years, digital forensics has relied on formats that often combined historical value with inconsistent semantics, underspecified interoperability, or implementation divergence.

CXF 1.0 represents a different direction:
- deterministic instead of ambiguous
- evidence-oriented instead of tool-oriented
- extensible without sacrificing core integrity
- suitable for open implementation and long-term archival use

## Immediate next steps

Now that CXF 1.0 is released, the focus shifts from architecture debate to ecosystem execution:
- publishing and maintaining the signed registry artifacts
- finalizing the normative CDDL package
- publishing interoperability results and coverage evidence
- supporting reference implementations and independent parsers
- organizing post-1.0 work for recovery evidence enrichment and future candidate suites

## Closing

CXF 1.0 is the result of sustained community discipline.

The core is stable. The evidentiary model is coherent. The governance path is clear enough to move from draft culture to implementation culture.

**CXF 1.0 is now released and ready for the broader community.**
